# План реализации приложения «Мини-анкета»

## 1. Инициализация репозитория

1.1. Создать локальный Git-репозиторий командой `git init`.

1.2. Сразу добавить корневой `.gitignore`. Исключить `node_modules/`, `dist/`, `coverage/`, `.env`, временные файлы редакторов и системные файлы.

1.3. Создать первый коммит с техническим заданием, планом и `.gitignore`.

1.4. Переименовать начальную ветку в `main`, подключить удалённый репозиторий `https://github.com/mangert/Otus_questions.git` как `origin` и опубликовать ветку командой `git push -u origin main`.

## 2. Техническая основа

2.1. Организовать монорепозиторий с npm workspaces: `backend/` и `frontend/`.

2.2. Выбрать стек: Node.js, TypeScript и Express для backend; React, TypeScript и Vite для frontend.

2.3. Подключить ESLint, Prettier и Vitest. Для проверки HTTP API использовать Supertest, для UI — React Testing Library.

2.4. Настроить корневые команды `npm run dev`, `npm run build`, `npm test` и `npm run lint`.

Планируемая структура:

```text
backend/src/        # API, вопросы, валидация и хранилище ответов
backend/tests/      # unit- и API-тесты backend
frontend/src/       # компоненты, API-клиент и стили
frontend/tests/     # тесты интерфейса
```

## 3. Контракт API

3.1. Описать общие TypeScript-типы вопросов, ответов и ошибок.

3.2. Зафиксировать `GET /questions`: ответ `200` и массив из 3–5 объектов `{ id, text }`.

3.3. Зафиксировать `POST /answers`: тело `{ answers: [{ questionId, value }] }`, успешное сохранение в памяти и ответ `201`.

3.4. Для пустых, неполных или некорректных данных возвращать `400` с объектом `{ error }`.

## 4. Реализация backend

4.1. Инициализировать TypeScript-проект и Express-приложение.

4.2. Добавить четыре жёстко заданных вопроса с уникальными постоянными `id`.

4.3. Реализовать маршруты `GET /questions` и `POST /answers`.

4.4. Вынести хранение отправок в отдельный модуль с массивом в памяти.

4.5. Реализовать проверку структуры запроса, существования `questionId` и непустого значения ответа.

4.6. Настроить JSON middleware, CORS для локальной разработки, единый обработчик ошибок и переменную `PORT`.

## 5. Unit-тесты backend

5.1. Протестировать получение набора вопросов и неизменность их идентификаторов.

5.2. Протестировать модуль хранения без запуска HTTP-сервера: сохранение отправки, получение снимка данных, защиту внутреннего состояния от внешних изменений и очистку хранилища. Пункт выполнен в рамках реализации п. 4.4.

5.3. Протестировать валидатор: корректный набор ответов, пустое тело, пустое значение, неизвестный и повторяющийся `questionId`.

5.4. Изолировать состояние между тестами с помощью очистки хранилища и не зависеть от порядка запуска.

## 6. Реализация frontend

6.1. Создать React-приложение и API-клиент с адресом backend из `VITE_API_URL`.

6.1.1. Использовать существующий каркас React, TypeScript и Vite в workspace `frontend` и сохранить точкой входа компонент `App`.

6.1.2. Добавить типизацию переменной окружения `VITE_API_URL`, прочитать её через `import.meta.env` и нормализовать базовый URL без завершающего `/`.

6.1.3. Создать независимый от React API-клиент и использовать в его публичных методах типы из пакета `@mini-survey/shared`.

6.1.4. Реализовать метод получения вопросов через `GET /questions`, проверять успешный HTTP-статус и возвращать `GetQuestionsResponse`.

6.1.5. Реализовать метод отправки `SubmitAnswersRequest` через `POST /answers` с заголовком `Content-Type: application/json`; успешный ответ `201` не разбирать как JSON.

6.1.6. Для неуспешных HTTP-статусов отклонять запрос с ошибкой, используя сообщение из ответа `ApiError`, когда оно доступно. Отображение ошибок и повтор запроса реализовать в п. 6.6.

6.1.7. Подготовить минимальную стилизацию на чистом CSS без UI-фреймворков: базовую типографику и фон, контейнер анкеты, поля и `label`, кнопки, сообщения, состояния `hover`, `focus-visible` и `disabled`, а также адаптацию для мобильных экранов.

6.1.8. Проверить frontend-командами сборки, линтера и форматирования, что каркас, типы, API-клиент и стили корректны. Загрузку данных в `App` оставить для п. 6.2.

6.2. Загрузить вопросы при открытии страницы и показать состояние загрузки.

6.3. Отрисовать обязательное текстовое поле с доступным `label` для каждого вопроса.

6.4. Хранить значения формы в состоянии и блокировать повторную отправку во время запроса.

6.5. Отправить данные через `POST /answers`, после успеха заменить форму сообщением «Спасибо!».

6.6. Обработать ошибки загрузки и отправки, добавить повтор действия и минимальную адаптивную вёрстку.

## 7. Тесты frontend

7.1. Подменить API-запросы и проверить состояния загрузки, ошибки и успешного получения вопросов.

7.2. Проверить отображение полей и изменение их значений пользователем.

7.3. Проверить структуру отправляемых ответов, блокировку повторной отправки и появление сообщения «Спасибо!».

7.4. Проверить доступность формы: связь `label` с полями и доступность сообщений об ошибках.

## 8. Общее тестирование

8.1. Добавить API-тесты через Supertest для обоих маршрутов и ответов `200`, `201` и `400`.

8.2. Выполнить полный набор проверок: `npm test`, `npm run lint` и `npm run build`.

8.3. Вручную пройти сквозной сценарий: открыть форму, заполнить поля, отправить ответы и увидеть «Спасибо!».

8.4. Проверить повторный запуск приложения и ожидаемую очистку данных из памяти после остановки backend.

## 9. Исправление замечаний аудита

9.1. Явно задать ограничение размера JSON-тела в `express.json`, сохранив небольшой лимит, достаточный для анкеты, и возвращать стандартный ответ `413 Payload Too Large` при его превышении.

9.2. Для `POST /answers` проверять `Content-Type`, возвращать `415 Unsupported Media Type` для запросов не в формате JSON и зафиксировать новое поведение в `API_CONTRACT.md`.

9.3. Вынести список разрешённых CORS origins в переменную `CORS_ORIGINS`, разобрать её в отдельной конфигурационной функции и оставить текущие локальные адреса fallback-значением.

9.4. Уточнить сообщения валидатора: указывать индекс элемента `answers`, в котором обнаружена ошибка, не делая точный текст сообщения частью публичного API-контракта.

9.5. Расширить Supertest- и unit-тесты: проверить некорректный JSON и статус `400`, превышение лимита и статус `413`, неподдерживаемый `Content-Type` и статус `415`, отсутствие сохранения отклонённых данных, а также разбор CORS-конфигурации и индекс элемента в ошибках валидатора.

9.6. Заменить в frontend конструкцию `.then(onFulfilled, onRejected)` безопасной цепочкой обработки промиса, устранить дублирование первичной и повторной загрузки вопросов и сохранить защиту от устаревших запросов и обновления размонтированного компонента.

9.7. Реализовать корректный жизненный цикл backend: сохранять экземпляр HTTP-сервера, обрабатывать `SIGTERM` и `SIGINT` через `server.close()`, обрабатывать ошибку запуска и завершать процесс с ненулевым кодом при фатальной ошибке.

9.8. Добавить тесты изменённого поведения frontend и backend, включая проверку безопасного завершения HTTP-сервера, не добавляя глобальный обработчик `unhandledRejection`, скрывающий источник ошибки.

9.9. После исправлений выполнить `npm test`, `npm run lint`, `npm run build` и проверку форматирования изменённых файлов.

## 10. Контейнеризация

10.1. Добавить multi-stage `Dockerfile` для backend: сборка TypeScript и запуск production-сборки в Node.js.

10.2. Добавить multi-stage `Dockerfile` для frontend: передать `VITE_API_URL=/api` как build argument, собрать Vite-приложение и раздавать статические файлы через Nginx.

10.3. Настроить Nginx как same-origin reverse proxy: направлять `/api/*` во внутренний backend-сервис с удалением префикса `/api`, чтобы браузер не зависел от имени Docker-сервиса и контейнерный запуск не требовал CORS.

10.4. Создать `docker-compose.yml`, объединяющий `backend` и `frontend` внутренней сетью: передать backend-переменные `PORT` и при необходимости `CORS_ORIGINS`, не публиковать backend наружу и опубликовать только порт frontend.

10.5. Добавить `.dockerignore` и `.env.example` с параметрами сборки и запуска контейнеров, отдельно пояснив, какие значения используются во время сборки frontend, а какие — во время запуска сервисов.

10.6. Проверить сборку и запуск командой `docker compose up --build`, доступность frontend и проксирование обоих API-маршрутов через `/api`.

10.7. Пройти сквозной сценарий в контейнерном окружении: открыть форму, загрузить вопросы, заполнить поля, отправить ответы и увидеть сообщение «Спасибо!».

10.8. Проверить корректное получение `SIGTERM`, остановку через `docker compose down`, освобождение опубликованных портов и ожидаемую очистку in-memory данных после повторного запуска.
