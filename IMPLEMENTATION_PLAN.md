# План реализации приложения «Мини-анкета»

## 1. Инициализация репозитория

1.1. Создать локальный Git-репозиторий командой `git init`.

1.2. Сразу добавить корневой `.gitignore`. Исключить `node_modules/`, `dist/`, `coverage/`, `.env`, временные файлы редакторов и системные файлы.

1.3. Создать первый коммит с техническим заданием, планом и `.gitignore`.

## 2. Техническая основа

2.1. Организовать монорепозиторий с npm workspaces: `backend/` и `frontend/`.

2.2. Выбрать стек: Node.js, TypeScript и Express для backend; React, TypeScript и Vite для frontend.

2.3. Подключить ESLint, Prettier и Vitest. Для проверки HTTP API использовать Supertest, для UI — React Testing Library.

2.4. Настроить корневые команды `npm run dev`, `npm run build`, `npm test` и `npm run lint`.

Планируемая структура:

```text
backend/src/        # API, вопросы, валидация и хранилище ответов
backend/tests/      # unit- и API-тесты backend
frontend/src/       # компоненты, API-клиент и стили
frontend/tests/     # тесты интерфейса
```

## 3. Контракт API

3.1. Описать общие TypeScript-типы вопросов, ответов и ошибок.

3.2. Зафиксировать `GET /questions`: ответ `200` и массив из 3–5 объектов `{ id, text }`.

3.3. Зафиксировать `POST /answers`: тело `{ answers: [{ questionId, value }] }`, успешное сохранение в памяти и ответ `201`.

3.4. Для пустых, неполных или некорректных данных возвращать `400` с объектом `{ error }`.

## 4. Реализация backend

4.1. Инициализировать TypeScript-проект и Express-приложение.

4.2. Добавить четыре жёстко заданных вопроса с уникальными постоянными `id`.

4.3. Реализовать маршруты `GET /questions` и `POST /answers`.

4.4. Вынести хранение отправок в отдельный модуль с массивом в памяти.

4.5. Реализовать проверку структуры запроса, существования `questionId` и непустого значения ответа.

4.6. Настроить JSON middleware, CORS для локальной разработки, единый обработчик ошибок и переменную `PORT`.

## 5. Unit-тесты backend

5.1. Протестировать получение набора вопросов и неизменность их идентификаторов.

5.2. Протестировать модуль хранения: добавление отправки и получение сохранённых данных без запуска HTTP-сервера.

5.3. Протестировать валидатор: корректный набор ответов, пустое тело, пустое значение, неизвестный и повторяющийся `questionId`.

5.4. Изолировать состояние между тестами с помощью очистки хранилища и не зависеть от порядка запуска.

## 6. Реализация frontend

6.1. Создать React-приложение и API-клиент с адресом backend из `VITE_API_URL`.

6.2. Загрузить вопросы при открытии страницы и показать состояние загрузки.

6.3. Отрисовать обязательное текстовое поле с доступным `label` для каждого вопроса.

6.4. Хранить значения формы в состоянии и блокировать повторную отправку во время запроса.

6.5. Отправить данные через `POST /answers`, после успеха заменить форму сообщением «Спасибо!».

6.6. Обработать ошибки загрузки и отправки, добавить повтор действия и минимальную адаптивную вёрстку.

## 7. Тесты frontend

7.1. Подменить API-запросы и проверить состояния загрузки, ошибки и успешного получения вопросов.

7.2. Проверить отображение полей и изменение их значений пользователем.

7.3. Проверить структуру отправляемых ответов, блокировку повторной отправки и появление сообщения «Спасибо!».

7.4. Проверить доступность формы: связь `label` с полями и доступность сообщений об ошибках.

## 8. Общее тестирование

8.1. Добавить API-тесты через Supertest для обоих маршрутов и ответов `200`, `201` и `400`.

8.2. Выполнить полный набор проверок: `npm test`, `npm run lint` и `npm run build`.

8.3. Вручную пройти сквозной сценарий: открыть форму, заполнить поля, отправить ответы и увидеть «Спасибо!».

8.4. Проверить повторный запуск приложения и ожидаемую очистку данных из памяти после остановки backend.

## 9. Контейнеризация

9.1. Добавить multi-stage `Dockerfile` для backend: сборка TypeScript и запуск production-сборки в Node.js.

9.2. Добавить multi-stage `Dockerfile` для frontend: сборка Vite и раздача статических файлов через Nginx.

9.3. Настроить Nginx как reverse proxy к backend, чтобы браузер обращался к API без зависимости от внутреннего имени Docker-сервиса.

9.4. Создать `docker-compose.yml`, объединяющий сервисы `backend` и `frontend`, с внутренней сетью, переменными окружения и опубликованным портом frontend.

9.5. Добавить `.dockerignore` и проверить запуск командой `docker compose up --build`.

9.6. Пройти сквозной сценарий в контейнерном окружении и проверить остановку через `docker compose down`.

## 10. Документация и критерии готовности

10.1. Добавить `.env.example` и `README.md` с требованиями, локальной установкой, тестированием и запуском через Docker Compose.

10.2. Обновить `AGENTS.md` фактической структурой, командами и соглашениями проекта.

10.3. Считать работу завершённой, когда локальный и контейнерный запуски документированы, API соответствует контракту, frontend проходит основной сценарий без перезагрузки, а тесты, линтер и сборка завершаются успешно.
