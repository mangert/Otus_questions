# Аудит кода — Мини-анкета

Проект: TypeScript-монорепозиторий (Express 5 + React + Vite). Проверены все исходные файлы `backend/src/`, `frontend/src/`, `shared/src/` и тесты.

---

## 1. Общие баги и риски

| # | Серьёзность | Описание | Файл |
|---|-------------|----------|------|
| 1 | **Высокая** | **Нет лимита тела запроса.** `express.json()` вызван без опции `limit` — злоумышленник может отправить гигантский JSON и вызвать OOM. | `backend/src/app.ts:18` |
| 2 | **Высокая** | **Express 5 + кастомный error handler.** Express 5 автоматически ловит ошибки из async-обработчиков, но в проекте используется классический `ErrorRequestHandler`. Если middleware до error handler выбросит исключение синхронно — поведение может отличаться от Express 4. | `backend/src/app.ts:37-63` |
| 3 | **Средняя** | **Нет проверки Content-Type на POST /answers.** Если клиент пошлёт `Content-Type: text/plain`, `express.json()` не распарсит тело, `body` будет `undefined`. Валидатор вернёт `"Тело запроса должно быть JSON-объектом"` — неочевидная ошибка. | `backend/src/app.ts:18` |
| 4 | **Средняя** | **CORS-origins не конфигурируются.** Жёстко зашиты `localhost:5173` и `127.0.0.1:5173`. При смене порта фронтенда или домена — CORS-ошибка. | `backend/src/app.ts:10-13` |
| 5 | **Низкая** | **`apiBaseUrl` вычисляется на момент загрузки модуля.** `import.meta.env.VITE_API_URL` читается один раз. Если переменная изменится (runtime), клиент не подхватит новое значение. | `frontend/src/api.ts:20-22` |
| 6 | **Низкая** | **Anti-pattern: `.then(onFulfilled, onRejected)` без цепочки `.catch()`.** Хотя коллбэки простые и не должны бросать, при ошибке в `onFulfilled` промис упадёт без обработки. | `frontend/src/App.tsx:35-52`, `59-80` |
| 7 | **Низкая** | **Нет конфигурации `engines` в workspace'ах.** Корневой `package.json` требует `node >=22.0.0`, но в `backend/package.json` и `frontend/package.json` это не продублировано — npm может не проверить версию при установке вложенных зависимостей. | `backend/package.json`, `frontend/package.json` |
| 8 | **Низкая** | **`shared/package.json` — exports только `types`, без `"default"`.** Некоторые старые резолверы могут не найти пакет, хотя для современных (Bundler, NodeNext) это корректно. | `shared/package.json:7-10` |
| 9 | **Низкая** | **Сообщение об ошибке не указывает, какое поле не прошло валидацию.** Валидатор возвращает общую фразу («Поля questionId и value должны быть строками»), не указывая индекс проблемного элемента. | `backend/src/validation.ts:33-46` |
| 10 | **Инфо** | **in-memory storage — данные теряются при перезапуске.** По спецификации это ожидаемо, но стоит упомянуть. | `backend/src/submissions.ts` |

## 2. Баги и риски для контейнеризации

### 2.1. Жёстко закодированные пути / адреса

| # | Описание | Файл | Исправление |
|---|----------|------|-------------|
| 1 | **CORS origins зашиты в `localhost:5173` / `127.0.0.1:5173`.** В контейнере фронтенд будет раздаваться через Nginx на порту 80 (или другом), и все CORS-запросы упадут. | `backend/src/app.ts:10-13` | Вынести в `process.env.CORS_ORIGINS` с fallback на текущий массив. |
| 2 | **Базовый URL API по умолчанию — `http://localhost:3000`.** В контейнере фронтенд и бэкенд — разные сервисы, localhost не сработает. | `frontend/src/api.ts:8` | Установить `VITE_API_URL` в docker-compose.yml. |

### 2.2. Проблемы с переменными окружения

| # | Описание | Файл | Статус |
|---|----------|------|--------|
| 1 | `process.env.PORT ?? 3000` — корректно, fallback есть. | `backend/src/index.ts:3` | ✅ ОК |
| 2 | `import.meta.env.VITE_API_URL ?? defaultApiUrl` — корректно, fallback есть. **Но:** Vite встраивает env-переменные на этапе **сборки**. Для разных окружений (dev/staging/prod) нужны разные образы или механизм runtime-конфигурации. | `frontend/src/api.ts:20-22` | ⚠️ Ограничение Vite |
| 3 | **CORS origins не конфигурируются через env** — `PORT` можно поменять, а `CORS_ORIGINS` — нет. | `backend/src/app.ts:10-13` | ❌ Нужен `process.env.CORS_ORIGINS` |
| 4 | **Нет `.env.example`** — хотя упомянут в плане (п. 10.1). | корень проекта | ❌ Отсутствует |

### 2.3. Обработка ошибок в асинхронном коде (в контексте контейнера)

| # | Описание | Статус |
|---|----------|--------|
| 1 | **Нет graceful shutdown.** При получении `SIGTERM` (контейнер стопится) сервер обрывает соединения мгновенно. Нужен `process.on('SIGTERM', ...)` с вызовом `server.close()`. | ❌ Отсутствует |
| 2 | **Unhandled promise rejection не обрабатывается.** Если где-то в цепочке промис будет rejected без обработчика, в Node >=15 процесс завершится с ненулевым кодом — контейнер перезапустится. | ❌ Нет `process.on('unhandledRejection', ...)` |
| 3 | **Логи — только `console.log`.** Это соответствует 12-factor app (stdout), но в контейнере нет структурированного логирования (JSON), что усложняет агрегацию логов. | ⚠️ Некритично |

### 2.4. Безопасность

| # | Описание | Файл | Комментарий |
|---|----------|------|-------------|
| 1 | **Нет лимита тела запроса** (повтор из п. 1.1). В контейнере без лимитов это может привести к OOM всего сервиса. | `backend/src/app.ts:18` | |
| 2 | **Нет rate limiting.** В production под нагрузкой контейнер может упасть от множества запросов. | N/A | |
| 3 | **Порт 3000 не настраивается для прода.** `PORT` через env настраивается, но в контейнере его надо пробрасывать явно. Для внутренней сети Docker можно оставить 3000, опубликовать — только через reverse proxy. | ⚠️ На усмотрение |
| 4 | **Helmet не используется.** Стандартные HTTP-заголовки безопасности (X-Content-Type-Options, X-Frame-Options и др.) не устанавливаются, хотя во frontend это менее критично (SPA). | N/A |
